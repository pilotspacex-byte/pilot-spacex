# Phase quick-260320-00i Plan 01: Member Management Bug Fixes Summary

**One-liner:** Fixed owner self-demotion security hole, frontend role case mismatches (422 errors), and added confirmation dialog for role changes.

## Objective

Fix 3 critical member management bugs and add 1 UX improvement:
1. **BUG-01 (Security):** Owner could change their own role via `PATCH /members/{userId}`, risking workspace lockout
2. **BUG-02 (422 errors):** Frontend sent lowercase roles to backend which requires UPPERCASE (schema validates `^(OWNER|ADMIN|MEMBER|GUEST)$`)
3. **BUG-03 (Broken display):** `useWorkspaceMembers` returned UPPERCASE roles but TypeScript type and all downstream consumers (role sorting, filtering, admin count, badge display) expected lowercase
4. **UX-05 (Safety):** Role changes executed immediately without confirmation

## Implementation Details

### Task 1: Backend Guard + Frontend Normalization (TDD)

**RED phase:** Added `TestOwnerSelfDemotion` class with 2 failing tests:
- `test_owner_cannot_demote_self`: verified `UnauthorizedError("Cannot change own role")` is raised
- `test_owner_can_change_other_member_role`: verified owner retains ability to change other members' roles

**GREEN phase:**

Backend — inserted guard in `update_member_role` after `actor_member` lookup, before `target_member` lookup:
```python
if payload.actor_id == payload.target_user_id and actor_member.is_owner:
    raise UnauthorizedError(
        "Cannot change own role. Use ownership transfer to reassign ownership."
    )
```

Frontend write path — `inviteMember` and `updateMemberRole` in `workspaces.ts` now call `.toUpperCase()` before sending to API.

Frontend read path — `useWorkspaceMembers` queryFn maps each member's role through `.toLowerCase()` to match the `'owner' | 'admin' | 'member' | 'guest'` TypeScript type.

### Task 2: Role Change Confirmation Dialog

Replaced the direct-execute `handleRoleChange` in `members-page.tsx` with a dialog-first pattern:
- Compares current vs. new role — no-op if same role selected
- Shows `ConfirmActionDialog` with member name, old role, new role
- Executes role change only on confirm (with toast feedback)
- Cancel dismisses without any state change

Uses existing `confirmDialog`/`closeConfirmDialog`/`setConfirmDialog` state already present in the component for `handleRemoveMember` and `handleCancelInvitation`.
